# 認証機能実装ドキュメント

## 概要

商品管理アプリに対して認証機能を追加した。

当初は誰でも商品管理画面へアクセスできる状態であり、ユーザーごとのデータ管理も行われていなかった。

今回の改善では、ログイン機能、新規登録機能、ログアウト機能を実装し、ユーザーごとに商品データを管理できる構成へ変更した。

また、単なる機能実装ではなく、将来的なクラウド連携やバックエンド連携を見据えた設計を意識して実装を行った。

---

# 実装前の課題

実装前は以下の課題が存在していた。

* 認証機能が存在しない
* 誰でも商品管理画面へアクセスできる
* ユーザー情報を保持していない
* ログイン状態を保持できない
* 商品データが全ユーザー共通である
* 利用者ごとのデータ管理ができない

そのため、実際のWebアプリケーションとしては不十分な状態であった。

---

# 実装内容

## 認証管理機能

認証処理を管理するために Custom Hook として useAuth を作成した。

### 実装機能

* ログイン
* 新規登録
* ログアウト
* 認証状態判定
* エラーメッセージ管理

認証ロジックをコンポーネントから分離することで、保守性と再利用性を向上させた。

---

## ログイン機能

ログイン画面を作成し、登録済みユーザー情報との照合を実施した。

### 実装内容

* メールアドレス入力
* パスワード入力
* 認証処理
* エラー表示

認証成功時のみ商品管理画面へアクセス可能とした。

---

## 新規登録機能

ユーザー自身でアカウント作成ができる機能を追加した。

### 入力項目

* ユーザー名
* メールアドレス
* パスワード

### 実装内容

* ユーザー登録
* 自動ログイン
* 重複メールチェック

---

## 入力バリデーション

不正なデータ登録を防止するためバリデーションを実装した。

### ログイン

* 必須チェック

### 新規登録

* 必須チェック
* メール形式チェック
* パスワード8文字以上
* 重複メールチェック

---

## localStorage対応

ブラウザをリロードした場合でも認証状態を維持するため localStorage を利用した。

### 保存情報

#### 登録ユーザー一覧

product-app-users

#### ログイン中ユーザー

product-app-login-user

### 実現したこと

* ページリロード後もログイン状態維持
* 登録ユーザー保持
* 再ログイン可能

---

## ユーザー別商品管理

商品データへ userId を追加した。

### Product型

```ts
{
  id: string;
  userId: string;
  name: string;
  completed: boolean;
}
```

### 実装内容

商品追加時にログインユーザーIDを保存し、商品一覧取得時にログインユーザーの商品だけを表示するように変更した。

### 効果

* ユーザーごとの商品分離
* 他ユーザーの商品非表示
* 実際のWebサービスに近い構成を実現

---

## UI改善

認証画面および商品管理画面のデザインを改善した。

### 改善内容

* グラデーション背景
* カードレイアウト
* モダンなフォームデザイン
* ホバーアニメーション
* レスポンシブ対応

### 効果

* 視認性向上
* 操作性向上
* Webサービスらしい見た目へ改善

# 技術選定理由

今回の認証機能実装では、現在の学習段階だけでなく、将来的な拡張性や実務での利用実績を考慮して技術選定を行った。

---

## React

フロントエンドフレームワークとしてReactを採用した。

### 選定理由

* 国内外での採用実績が豊富
* 求人市場での需要が高い
* コンポーネント指向による高い再利用性
* AWS AmplifyやAWS Cognitoとの親和性が高い
* 将来的にReact Nativeへの展開も可能

### 採用しなかった技術

#### Vue.js

学習コストが低く開発速度も高いが、今後習得を目指しているReactおよびAWSとの親和性を優先した。

#### Angular

大規模開発向けの機能が充実しているが、本アプリケーション規模ではオーバースペックと判断した。

---

## TypeScript

JavaScriptではなくTypeScriptを採用した。

### 選定理由

* コンパイル時に型エラーを検出できる
* IDE補完が強力
* リファクタリングが容易
* Reactとの組み合わせが一般的

認証機能ではユーザー情報や商品情報など複数のデータ構造を扱うため、型安全性のメリットが大きいと判断した。

### 採用しなかった技術

#### JavaScript

学習コストは低いが、規模拡大時の保守性や品質維持を考慮しTypeScriptを選択した。

---

## Custom Hook（useAuth）

認証処理を管理するためにCustom Hookを採用した。

### 選定理由

* UIと認証ロジックを分離できる
* 再利用性が高い
* テストしやすい
* 将来的な認証基盤変更に強い

### 採用しなかった実装方法

認証ロジックをコンポーネントへ直接実装する方法もあるが、保守性や拡張性を考慮して採用しなかった。

### 将来的なメリット

現在はlocalStorage認証を利用しているが、将来的にAWS CognitoやSpring Boot認証へ移行する際も、useAuth内部の変更のみで対応できる。

---

## React Query

API通信管理ライブラリとしてReact Queryを採用した。

### 選定理由

* キャッシュ管理機能
* 自動再取得機能
* API通信状態管理機能
* Reactとの親和性

### 採用しなかった技術

#### useEffect + useState

小規模な実装であれば十分対応可能である。

しかし、

* ローディング管理
* エラーハンドリング
* キャッシュ管理

を個別実装する必要があり、保守コストが高くなるため採用しなかった。

#### Redux Toolkit Query

大規模システムでは有力な選択肢である。

ただし、本アプリケーションではReduxを導入していないため、React Queryの方がシンプルであると判断した。

---

## localStorage

認証情報保存先としてlocalStorageを採用した。

### 選定理由

* バックエンド不要
* 学習コストが低い
* ログイン状態保持を実現できる
* 認証フロー学習に適している

### 採用しなかった技術

#### Session Storage

ブラウザ終了時に情報が消えるため採用しなかった。

#### Cookie

本番環境では有力な選択肢であるが、学習段階では実装コストが高いため採用しなかった。

#### JWT

本来はこちらが望ましい。

ただしバックエンドが必要になるため、今回の学習範囲からは除外した。

---

## Vitest

テストフレームワークとしてVitestを採用した。

### 選定理由

* Viteとの統合が容易
* Jest互換APIを利用できる
* 実行速度が速い
* TypeScriptとの相性が良い

### 採用しなかった技術

#### Jest

実績は豊富である。

しかしVite環境ではVitestの方が設定がシンプルであり、開発体験も良いため採用した。

---

## Testing Library

UIテストライブラリとして採用した。

### 選定理由

* ユーザー操作を重視したテスト
* React公式でも広く利用されている
* 実際の利用シナリオに近いテストが可能

### 採用しなかった技術

#### Enzyme

コンポーネント内部実装への依存が強く、現在はメンテナンスも活発ではないため採用しなかった。

---

# セキュリティに対する考え方

今回の認証機能は学習用実装として構築している。

そのため、パスワードは簡易的に保存しているが、本番運用を想定した場合には改善が必要であると認識している。

将来的には以下の対応を行う予定である。

* AWS Cognitoによる認証管理
* JWT認証
* パスワードのハッシュ化
* メール認証
* パスワードリセット機能
* 多要素認証（MFA）

今回の実装では認証処理の流れや設計方針の理解を優先した。
